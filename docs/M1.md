# Milestone 1 — HRF Arcs engine on the JVM

**Goal:** compile the vendored HRF Arcs rules engine for the JVM and run a full
Arcs game headless (no browser), so the bot can drive it. This is the only
high-risk milestone; everything else builds on it.

The sources are already in the repo under **`hrf-engine/`** (vendored from
upstream HRF, MIT-licensed — see `hrf-engine/LICENSE`). The next session does
**not** need to fetch anything; M1 is build configuration + iteration.

## What's vendored

- `hrf-engine/*.scala` — 40 top-level files: the `hrf` framework/core
  (`base`, `serialize`, `meta`, `colmat`, `options`, `compute`, `elem`,
  `host`, `journal`, `timeline`, the `*-jvm.scala` shims, the generic `bot` AI,
  etc.) **plus** browser/UI files we exclude from the JVM build.
- `hrf-engine/arcs/*.scala` — 45 files: the full Arcs + Blighted Reach game
  (`game`, `meta`, `serialize`, the 28 `fate-*`, blight, leaders, lore, summit,
  guilds, setup…). Only `arcs/ui.scala` and `arcs/styles.scala` are
  browser-bound and excluded.

## Why this compiles on the JVM (the analysis)

- **License: MIT.** Vendoring is fine; keep the notice.
- **Arcs rules are JVM-clean.** Of 45 `arcs/` files only `ui.scala`/`styles.scala`
  touch the DOM. All rules logic is portable.
- **`elem.scala` (the rich-text `Elem` type Arcs uses for move/question text)
  has no DOM dependency.** This was the main risk and it's clear — we can read
  `UserAction.option`/`question` text on the JVM without a browser.
- **HRF already ships JVM cross-compile shims.** `base.scala` uses
  `@scalajs.reflect.annotation.EnableReflectiveInstantiation` and
  `serialize.scala` uses `import scalajs.reflect._`; **`reflect-jvm.scala`
  declares `package scala.scalajs.reflect.*` to provide these on the JVM.**
  Likewise `log-jvm`, `host-jvm`, `grey-jvm`, `timeline-jvm` mirror the JS ones.
  So the cross-compile recipe is: keep `*-jvm.scala` + `host.scala`, drop the
  JS/DOM files. (This is the mirror image of upstream `build.sbt`'s
  `excludeFilter`, which excludes the `*-jvm` files for the JS target.)

## The build wiring (already in `build.sbt`)

`hrfEngine` project:
- `Compile / scalaSource := baseDirectory.value` so the top-level `*.scala` and
  the `arcs/` subdir both compile (recursively under `hrf-engine/`).
- `excludeFilter` drops the browser/Scala.js sources (basename match, so it also
  removes `arcs/ui.scala` + `arcs/styles.scala`).
- JVM library deps: `fastparse`, `pprint`, `fansi`, `blake3`,
  `scala-collection-contrib`, `scala-parallel-collections` (the last for
  `host.scala`'s `scala.collection.parallel.CollectionConverters`).
- Scala `2.13.16` (matches upstream HRF).

`hrfEngine` is intentionally **not** aggregated into `root` or depended on by
`engineBridge` yet, so the existing `sbt compile` / CI stays green while you work
M1 in isolation.

## Steps

1. **Compile the engine in isolation:**
   ```
   sbt hrfEngine/compile
   ```
   Iterate on the `excludeFilter` in `build.sbt` for any straggler browser/JS
   file the compiler flags (all sources are present, so this is pure filter
   tuning — no fetching). Candidates that may or may not need excluding depending
   on cross-deps: `grey-jvm.scala`, `bot.scala`, the `*-tracker.scala` family.
2. **Verify action reflection on the JVM.** `Serialize.parseExpr` resolves
   action class names via `scalajs.reflect` → the `reflect-jvm.scala` shim
   (`serialize.scala`). Confirm `arcs.*` action case classes instantiate by name
   on the JVM (write a tiny round-trip test: `parseAction(write(a)) == a` for a
   sample action).
3. **Headless self-play.** Reproduce `arcs/host.scala` / `BaseHost.main`
   (upstream `host.scala`): build an `arcs.Game` via `MetaBR`'s start action,
   loop `performContinue`, auto-resolve `Roll`/`Shuffle`/`Random`, use the
   generic `bot` AI (or random) for `Ask`, play start→finish, print the journal.
   This proves the engine runs off-browser.
4. **Wire it in.** Once `hrfEngine/compile` is green and self-play works:
   - add `hrfEngine` to `root.aggregate`,
   - `engineBridge.dependsOn(hrfEngine)`,
   - uncomment the HRF imports / `/* HRF: ... */` bodies in
     `modules/engine-bridge/.../EngineSession.scala`,
   - optionally add a CI job for `sbt hrfEngine/compile`.

## Done when

`sbt hrfEngine/compile` is green **and** a headless CLI plays a full Arcs:
The Blighted Reach game to completion, emitting a journal of action lines that
round-trips through `Serialize`. That's the foundation M2 (journal/session) and
M4 (Discord) build on.

## Note on keeping vendored sources in sync

These are a point-in-time copy of HRF `v0.8.140`. To pull upstream fixes later,
re-copy `hrf-engine/` from the HRF source (or migrate to a git submodule if a
public HRF repo URL is available). Do **not** edit the vendored rules logic;
keep changes to build/packaging only, so re-syncing stays trivial.
